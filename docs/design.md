## Problem Statement

Germany's energy transition plans are evolving in real time under the current Bundesregierung. Federal Minister Katharina Reiche has signaled material departures from the previous Klimaneutral-2045 trajectory: expanded gas backup (Kraftwerksstrategie), a delayed hydrogen ramp, slowed renewables pace, and reframed targets. These statements shape national policy and billions in capital allocation, but there is currently no public, methodologically rigorous tool that translates the evolving plan into runnable numbers and stress-tests it against bad weather years.

Researchers (Fraunhofer ISE, TU Berlin, Agora Energiewende) do this work internally with closed models. Public viz tools (Energy-Charts.info, Bundesnetzagentur SMARD) show historical and live data, not counterfactual futures. Journalists writing about Reiche's plan have no shared reference point. Energy committee staff debate without a common simulator.

The gap is a public, open-source, web-accessible case study tool that:
1. Translates published policy plans (Reiche-era, Habeck-era, academic alternatives) into versioned, source-cited scenario data.
2. Runs hour-by-hour optimal dispatch under each scenario against historical weather years.
3. Surfaces stress hours, regional bottlenecks, storage demand, and curtailment.
4. Lets anyone — researcher, journalist, civil servant, energy nerd — see and share the results.

## What Makes This Cool

Three things stack:

1. **Timeliness.** Reiche's plan is being shaped right now in 2026. No one else has it modeled publicly. First-mover advantage isn't startup-marketing flavored here; it's "the canonical reference everyone links to."
2. **Political weight without polemic.** A defensible, methodologically rigorous tool that simply *runs the official plan* produces conversations no opinion piece can. The numbers speak.
3. **Scenario revisioning as a first-class feature.** Reiche's plan moves; the tool moves with it. Every scenario is dated, versioned, source-cited. Compare any two versions side by side. The "moving target" problem becomes a feature, not a bug.

The signature contribution is the case study, not the engine. PyPSA-Eur already exists; nobody has done the work of translating the current Bundesregierung's evolving plan into a versioned, runnable scenario and stress-testing it publicly.

## Constraints

- **Author bandwidth**: evenings/weekends, async with AI coding partner. No calendar pressure, but realistic effort budget; scope must remain tractable.
- **Hosting**: must deploy on Vercel from V0. Live optimization in browser is infeasible — modeling runs offline, results are pre-computed and served as static data.
- **Methodology**: research-grade rigor non-negotiable. Every assumption in every scenario must trace to a citable source.
- **Visual quality**: must look polished from V0, not "academic demo" aesthetic. Pre-built UI components are explicitly allowed (shadcn/ui, Tailwind) to keep build cost low.
- **Hydrogen modeling**: realistic but not headline. Author has BSc-level depth on hydrogen electrolysis with time-resolved renewables and utilization rate; this depth gets used inside the general model rather than turned into the marketing angle.
- **Single-author project at start**, but architected such that a future contributor could add scenarios via PR without engine knowledge.

## Premises (locked)

1. Research-grade public artifact AND a polished, showable website — both.
2. Stand on PyPSA-Eur / Linopy / atlite / ERA5; do not rebuild optimization machinery.
3. No fixed time budget; async with AI; ship when shippable.
4. Methodological rigor + nice design from V0. Pretty + reproducible.
5. Distribution: hosted website on Vercel + open-source GitHub repo + cold-email to research orgs and journalists.
6. Signature contribution = stress-test of the Bundesregierung's currently-evolving plan vs. alternative scenarios. Hydrogen modeled realistically inside the general model, not as the headline.
7. Scenarios are versioned, dated, source-cited artifacts. The tool is designed to keep working as plans evolve over years.

## Approaches Considered

### Approach A: PyPSA-Eur fork + Next.js viz, pre-computed runs (CHOSEN)
Fork PyPSA-Eur, author scenarios as versioned YAML/JSON configs, run Snakemake pipeline offline, commit Parquet results to repo (or LFS), Next.js frontend reads results and visualizes. Vercel hosts. Adding/updating a scenario = update config, re-run pipeline, commit results, Vercel auto-redeploys.

**Pros**: Mature optimization engine; defensible methodology; compatibility with the research community (potential upstream PyPSA-Eur contribution); Vercel-friendly architecture; modular scenario files map naturally to versioning requirement; author's BSc thesis on hydrogen plugs into PyPSA-Eur's hydrogen components.

**Cons**: Pipeline runs offline only — no live re-runs from the browser. Adding scenarios is author-mediated (or requires a contributor with PyPSA-Eur knowledge). PyPSA-Eur has a learning curve.

**Reuses**: PyPSA-Eur, atlite (weather→power), linopy (LP solver), ERA5 reanalysis (weather data), Snakemake (pipeline), Next.js, shadcn/ui, Tailwind, Recharts (or Visx/Observable Plot), Vercel.

### Approach B: Custom optimization in linopy from primitives
Same frontend, but author writes the LP from scratch in linopy rather than via PyPSA-Eur.

**Pros**: Deeper learning of optimization stack; full control over storage and hydrogen formulations.

**Cons**: Much slower to first useful result; more modeling bugs; harder to validate against literature; worse compatibility with research community.

**Verdict**: Rejected. The artifact is the goal; PyPSA-Eur is the right shoulders to stand on. Author can still customize hydrogen components within PyPSA-Eur where their BSc work goes beyond the defaults.

### Approach C: Heuristic dispatch in-browser (live, no LP)
Skip the LP entirely. Implement merit-order dispatch + simple storage rules as TypeScript that runs live in the browser.

**Pros**: Fully interactive; no pre-compute; maximum demo polish.

**Cons**: Not methodologically defensible. Researchers will not cite a heuristic. Violates premises #1 and #4.

**Verdict**: Rejected. The premise was "research-grade citable artifact." This would make it a viz toy.

## Recommended Approach

**Approach A.** Honors all seven premises. Lowest engine risk (PyPSA-Eur is battle-tested). Highest compatibility with target audiences (researchers and journalists). Scenario versioning maps cleanly onto config-file-as-artifact pattern. Frontend technology stack matches "looks nice from V0" requirement without inventing visual design from scratch.

### Scenario schema (versioned, source-cited)
Each scenario is a YAML file with strict Pydantic-validated schema:

```yaml
id: reiche
version: "2026-05-17.0"          # ISO date + patch component (.0, .1) for non-substantive edits
display_name: "Reiche Plan (Stand 2026-05)"
description: |
  Translation of BMWK statements and Koalitionsvertrag positions
  as of 2026-05-17 into runnable scenario data. Conservative
  renewables pace, expanded gas backup, delayed hydrogen ramp.
authors: [matthis-damm]
sources:
  - {ref: "BMWK-2026-04-12", title: "Reiche Pressemitteilung Kraftwerksstrategie", url: "..."}
  - {ref: "Koalitionsvertrag-2025", title: "Koalitionsvertrag CDU/CSU-SPD § 7", url: "..."}
  - {ref: "Reiche-Bundestag-2026-03-18", title: "Bundestagsrede", url: "..."}
capacities_2035_gw:
  wind_onshore:
    50hertz: 18.2
    tennet: 22.4
    amprion: 14.1
    transnetbw: 4.3
    citation_ref: "BMWK-2026-04-12"
  wind_offshore: { ... }
  solar_pv: { ... }
  gas_backup: { ... }
  hydrogen_electrolyzer: { ... }
  hydrogen_storage_twh: { ... }
  battery_storage_gwh: { ... }
  pumped_hydro: { ... }
demand_2035:
  baseline_twh: 540
  heat_pump_share: 0.42
  ev_share_passenger: 0.55
  electrolyzer_demand_twh: ...
  citation_ref: ["BMWK-Langfristszenarien-2024", "Reiche-Bundestag-2026-03-18"]
transmission_ntc_gw:
  # Inter-ÜNB-zone net transfer capacities, by direction
  50hertz_to_tennet: 12.5
  tennet_to_amprion: 8.7
  ...
# NOTE: Per-scenario history (citation fixes, restated numbers) lives in
# scenarios/{family}/CHANGELOG.md, NOT inline here. Git log is the authoritative
# record of file changes; the CHANGELOG.md summarizes substantive movements.
```

**Versioning rules:**
- Substantive change (capacity, demand, NTC, storage) → bump version to next ISO date (`2026-09-15.0`), keep old file.
- Non-substantive change (citation URL fix, typo, description rewording) → bump patch (`2026-05-17.0` → `2026-05-17.1`), same file edited in place; CHANGELOG.md notes the fix.
- Patch bumps do NOT invalidate Snakemake cache (only the substantive fields are hashed for `--rerun-triggers params`).

### Versioning model
- Each scenario file is versioned `YYYY-MM-DD.PATCH` (e.g., `2026-05-17.0`).
- Substantive change (capacities, demand, storage, NTC) → new ISO-date file, e.g., `reiche/2026-09-15.0.yml`. Previous version stays in repo.
- Non-substantive change (citation fix, typo, description) → bump patch component on the same file. Per-family CHANGELOG.md tracks the substantive history.
- Index file `scenarios/index.yml` lists all scenario families with metadata.
- Frontend shows all versions of a scenario family with a date selector; comparison view can show any two versions side by side.
- Scenarios point to their successor/predecessor via `supersedes`/`superseded_by` fields.

### Translation layer
A Python module under `modeling/translation/` reads scenario YAML, validates against a Pydantic schema, and emits the PyPSA-Eur config files plus per-technology overrides needed for a run. PyPSA-Eur input files are treated as generated artifacts (in `.gitignore`); scenarios are the single source of truth. The translation layer is the natural place for citation-enforcement: refuse to translate if substantive fields lack `citation_ref`. Translation logic is unit-tested independently of the LP solve.

### Pipeline (Snakemake)
- Inputs: weather year YAML + scenario YAML (validated through translation layer).
- Steps:
  1. Translate scenario → PyPSA-Eur config (Python translation module).
  2. Weather conversion: atlite produces hourly capacity factors from ERA5 reanalysis under the scenario's 2035 fleet.
  3. PyPSA-Eur model build using hand-curated 4-zone busmap (`modeling/busmap/unb_busmap.csv`) — k-means clustering is NOT used; the busmap is hand-mapped to ÜNB Regelzonen with inter-zone NTC values from BNetzA Monitoring reports.
  4. linopy LP solve via HiGHS (free, default). Solver abstracted so Gurobi can be swapped in later if solve time becomes painful.
  5. Hydrogen storage uses PyPSA-Eur's Links + Stores scaffolding with BSc-thesis-grade efficiency curves overriding the defaults (full 8760-hour SoC, no weekly aggregation).
  6. Hourly Parquet results PLUS pre-aggregated daily and weekly summaries written for each (scenario × weather-year).
  7. All result files pushed to R2 bucket; bucket manifest JSON updated and committed to repo.
- Invalidation: `--rerun-triggers params input` keyed on the scenario `version` field. Patch bumps do not invalidate; ISO-date bumps do. Code changes invalidate via Snakemake's `code` trigger.
- Runs offline on author's machine. CI runs a 2-zone × 168-hour smoke variant in <2 min on PRs.

### Result storage and frontend data delivery
- **Storage**: Parquet (hourly + daily + weekly aggregates) lives in a Cloudflare R2 bucket. R2 is ~€0-5/month at this scale, zero egress fees. A `scenarios-manifest.json` committed in the repo maps `(scenario_family, version, weather_year, resolution)` → bucket URL.
- **Frontend access**: Next.js reads the manifest at build time, fetches pre-aggregated JSON at request time. Overview views use daily or weekly summaries (small payload, fast render); drill-down to hourly fetches the hourly Parquet-derived JSON only on user zoom. Naive Parquet-in-browser is NOT used in V0 (DuckDB-WASM remains a V1 upgrade path).
- **Stack**: Next.js 15 App Router + TypeScript + Tailwind + shadcn/ui + Recharts.
- **Pages**:
  - `/` — landing: project mission, scenario chooser, one big demo run.
  - `/scenarios` — list all scenario families with version history and citations.
  - `/run/[scenario]/[version]/[weather-year]` — full run view: stacked-area generation per zone, cross-zone flow timeline, storage SoC, stress timeline, downloadable raw Parquet link.
  - `/compare` — pick any two scenarios or any two versions of one scenario, side-by-side under same weather year.
  - `/methodology` — how scenarios are built, all citations.
- **Visualization choices**: Recharts for the main timeline (good Vercel performance, theme integrates with shadcn); Visx for the cross-zone flow Sankey-ish view if Recharts is insufficient.

### CI/CD
GitHub Actions workflow `.github/workflows/ci.yml` runs on every PR/push to main:
- Python: `ruff check`, `ruff format --check`, `pytest` (translation layer + schema + hydrogen efficiency tests).
- JS: `eslint`, `tsc --noEmit`, `pnpm build`.

## Risks and Mitigations

- **Risk: Reiche's plan is hard to pin down (declared, not published).**
  Mitigation: Strict citation discipline. Every number traces to a dated source. When sources are ambiguous, document the interpretation explicitly in `methodology.md` and offer alternative-interpretation scenarios where useful.

- **Risk: Scenario data ages.**
  Mitigation: Versioning model makes this a feature. Old versions stay forever and can be re-run; new versions ship as new files.

- **Risk: Political reading regardless of methodology.**
  Mitigation: Tone of presentation is rigorous and neutral. Methodology page leads. Comparisons are always against multiple alternatives, never Reiche-vs-nothing. No editorializing in copy. Results speak for themselves.

- **Risk: PyPSA-Eur learning curve eats more time than expected.**
  Mitigation: V0 starts with vanilla PyPSA-Eur configuration; BSc-grade hydrogen customizations can ship in V0 only if they fit cleanly via PyPSA-Eur's override hooks, else deferred to V1. Better to ship a less-bespoke V0 than to die in customization.

- **Risk: Hydrogen part-load efficiency is NOT a config knob in PyPSA-Eur.**
  Discovered during build step 2 (see [methodology.md § 7](methodology.md#7-hydrogen-system--the-honest-version)). PyPSA-Eur models the electrolyzer as a `Link` with a single constant efficiency fetched from the costs CSV; `p_min_pu` only clips the operating range, not the efficiency curve. The BSc-grade override is a real modeling extension to `prepare_sector_network.py`, not a configuration tweak. V0 plan: piecewise-linear efficiency via N parallel Links per electrolyzer instance. Falls back to constant efficiency if the piecewise extension proves too time-consuming, with a clear note on the methodology page about what was simplified.

- **Risk: Inter-zone NTC values do not match BNetzA-published numbers.**
  PyPSA-Eur computes inter-cluster transmission capacity by summing the underlying physical lines that cross cluster boundaries. The result will diverge from the politically-quoted BNetzA NTC values. V0 decision: accept PyPSA-Eur's line sums and disclose the discrepancy on the methodology page. BNetzA-cited NTC override remains a V1 option.

- **Risk: External-data dependencies (ERA5 weather, costs CSV) break reproducibility.**
  PyPSA-Eur downloads ERA5 cutouts from Copernicus (requires CDS API token) and costs CSV from data.pypsa.org at pipeline run time. CI cannot rely on these. Mitigation: commit a small frozen cutout fixture + `costs_2035.csv` under `modeling/tests/fixtures/` for the mini E2E run. Production runs use the live downloads.

- **Risk: Result data outgrows git.**
  Mitigation: Results live in R2 bucket from V0 (Git LFS not used). Repo only contains the manifest pointing to bucket URLs. Per-scenario typical size: ~50-200 MB hourly Parquet; trivial for R2 at this scale.

- **Risk: LP solve time on HiGHS becomes a bottleneck.**
  Estimate: ~15-45 min per run for 4 zones × 8760 hours × ~8 technologies. 9 runs serial ≈ 3-5 hours one-time. Tolerable for batched offline runs. Mitigation: solver is abstracted — Gurobi academic license can be swapped in if iteration becomes painful, drops total to ~30 min.

- **Risk: Frontend perf with 8760-hour data per scenario per zone.**
  Mitigation: Pipeline pre-aggregates every run to daily and weekly summaries alongside hourly Parquet. Frontend overview views never load hourly data; drill-down fetches hourly only on user zoom. Naive payload is ~5-10 MB per scenario gzipped JSON; pre-aggregated summaries are <1 MB.

- **Risk: PyPSA-Eur upgrade silently changes dispatch results.**
  Mitigation: Golden-output regression test locks the V0 baseline run's Parquet checksum. Any future PyPSA-Eur upgrade that changes numbers fails CI and requires intentional unlock with an explanation. This is the highest-value test in the suite.

- **Risk: Hand-curated 4-zone busmap diverges from BNetzA reality.**
  Mitigation: NTC values cited per direction with BNetzA Monitoring report references; the busmap CSV is reviewable in PR. If significant grid topology changes occur, busmap gets a version bump and CHANGELOG entry.

- **Risk: "Slop" — assumptions chosen for narrative.**
  Mitigation: Author commits to source citations per assumption. Translation layer refuses to compile scenarios with missing citations. Methodology page is checked by external reviewer (academic contact, e.g., someone from PyPSA-Eur team) before public launch.

## Failure Modes (per new codepath)

For each new codepath, a realistic production failure scenario, and what catches it:

| Codepath | Realistic failure | Test? | Error handling? | User experience? |
|---|---|---|---|---|
| `translation/translate.py` | Scenario YAML references undefined ÜNB zone (typo: "tennett") | ✓ Pydantic schema test | ✓ Validation error before run | Clear error in build log; no silent miscompute |
| `translation/translate.py` | Substantive field missing `citation_ref` | ✓ Translation test | ✓ Translation refuses; build fails | Clear error; forces citation discipline |
| `hydrogen/efficiency_curves.py` | Override doesn't actually attach to PyPSA-Eur components (typo in component name) | ✓ Benchmark test (compares solve output against expected efficiency) | ✗ Solver silently uses default | Test catches it; without test would silently use wrong physics |
| `Snakefile` mini E2E in CI | Solver doesn't converge in CI's tiny model | ✓ E2E smoke test | ✓ CI fails on solver error | PR blocked; addressed by tightening test scenario |
| `web/lib/data-loader.ts` | R2 manifest references missing Parquet | ✓ Data-loader test | ✓ Graceful fallback to "data unavailable" UI | User sees explicit error message, not infinite spinner |
| `web/lib/data-loader.ts` | Schema mismatch (R2 file has columns frontend didn't expect) | ✓ Snapshot test | ✓ Typed error caught at parse | User sees explicit error |
| Golden regression | PyPSA-Eur upgrade changes default cost data → dispatch shifts | ✓ Golden-output checksum | ✓ CI fails | PR blocked until intentional unlock with explanation |

No silent failures in V0 codepaths. The `hydrogen/efficiency_curves.py` benchmark test is the most subtle: without it, the BSc-grade override could silently fail to bind and the model would run on stock defaults, undermining the signature contribution.

## Test Plan (V0)

For a citable research artifact, the test plan is part of the architecture. Every path below is required for V0; deferred tests are listed under V1.

### Required for V0

1. **Schema validation tests** (`modeling/translation/tests/test_schema.py`)
   Pydantic-driven. Each invariant in the scenario schema gets a unit test: zone name must be one of `{50hertz, tennet, amprion, transnetbw}`; substantive fields must have `citation_ref`; version must match `YYYY-MM-DD.N` pattern; capacities are non-negative.

2. **Translation layer tests** (`modeling/translation/tests/test_translate.py`)
   - `test_valid_scenario_translates`: known-good Reiche scenario produces expected PyPSA-Eur config.
   - `test_missing_citation_ref_fails`: substantive field without citation raises clear error.
   - `test_invalid_zone_name_fails`: typo in zone name raises clear error before any solve attempt.
   - `test_version_mismatch_fails`: scenario `version` not matching `scenarios/index.yml` raises clear error.

3. **Version-hash test** (`modeling/translation/tests/test_versioning.py`)
   - `test_substantive_content_same_hash`: edit only `citation_ref` URL → version hash unchanged → Snakemake cache hit.
   - `test_capacity_change_new_hash`: edit `wind_onshore.tennet` → version hash changes → Snakemake re-runs.

4. **Hydrogen efficiency benchmark** (`modeling/hydrogen/tests/test_efficiency_curves.py`)
   Critical: without this, the BSc-grade override could silently fail to bind. Compare model output against a published or analytically-derived electrolyzer part-load efficiency curve. Tolerance: ±1%.

5. **Mini end-to-end smoke test** (CI-gated)
   2-zone × 168-hour PyPSA-Eur run with a frozen tiny scenario. Asserts the pipeline produces non-empty Parquet output. Target runtime: <2 min on GitHub Actions free tier.

6. **Frontend data-loader tests** (`web/lib/__tests__/data-loader.test.ts`)
   - `test_valid_json_parses`: typed result for known-good payload.
   - `test_missing_scenario_graceful`: data-loader returns explicit "unavailable" state, not exception.
   - `test_schema_mismatch_typed_error`: payload with unexpected columns raises typed error caught by UI.

7. **CRITICAL — Golden-output regression test** (`modeling/tests/test_golden_baseline.py`)
   Locks the V0 baseline run's output checksums for one canonical scenario (e.g., `reiche/2026-05-17.0` × weather year 2010). Future PyPSA-Eur upgrades, dispatch-logic changes, or busmap edits that alter numbers will fail CI. Unlocking requires bumping a `golden_baseline_version` constant and writing one sentence in the test's docstring explaining why. This is the single highest-impact test in the suite: it converts silent semantic regressions into noisy failed CI runs.

### Deferred to V1

- Property-based tests on dispatch (e.g., "storage SoC never goes negative") via Hypothesis — useful for catching subtle model bugs, but the golden test covers most regressions in V0.
- Frontend behavior tests via Playwright — V0 frontend is mostly chart rendering; snapshot + data-loader tests suffice.
- Multi-version comparison sanity tests (`reiche/2026-05-17.0` vs `reiche/2026-09-15.0` should produce monotonic shifts in expected directions) — depends on V1 multi-version data being available.

### Coverage target
V0 ships with 11 test paths covered. Coverage diagram is maintained in `docs/methodology.md` and updated when codepaths change.
